# 采样设计与覆盖模板

这份文档专门回答一个问题：

> 如果 `saoshu` 的 `sampled` 不是纯技术抽样，而是贴近扫书社区真实习惯的“决策导向抽查”，那它应该怎么设计？

本文件不是当前代码的实现说明，而是后续把社区习惯、产品语义与执行层行为对齐的设计草案。

## 设计背景

当前已知的社区习惯并不是“均匀随机抽样”，而更接近：

- 先看前 100 章左右，确认早期观感与主要风险
- 首尾各看一部分，防结局翻车与后期关系反转
- 长书只在必要时补中段或关键大篇章，而不是平均每段都扫一点

因此，`sampled` 更适合被理解为：

- **决策导向摸底**
- 而不是 **统计意义代表性抽样**

## 设计目标

- 让 `sampled` 的用户语义更接近真实扫书习惯
- 让报告明确说明“扫了哪里、没扫哪里、为什么这么扫” 
- 让内部抽样逻辑可以逐步演进，而不必不断新增用户可见模式名
- 为未来 `chapter-full` / `full-book` 的升级路径提供清晰对照

## 非目标

- 不是现在就新增一堆新的用户可见模式
- 不是现在就废弃 `economy / performance`
- 不是把所有抽查模板都立刻写死进 CLI 主入口
- 不是追求数学抽样最优，而忽视用户真实决策行为

## 用户层与实现层边界

### 用户层

用户首先应理解：

- `sampled`：快速摸底
- `chapter-full`：章节级尽量完整
- `full-book`：整书最终确认

### 实现层

实现层可以在 `sampled` 内部继续细分不同的抽查模板，但这些模板默认不应全部直接暴露为新的主模式名。

换句话说：

- `sampled` 是用户可理解的产品层概念
- `opening-100`、`head-tail` 这类是内部 coverage template / sampling profile

## 核心原则

### 1. 先服务决策，再服务均匀性

扫书用户最关心的是：

- 早期是否能看下去
- 后期是否翻车
- 是否有一票否决级雷点

因此抽样优先级应围绕这些问题，而不是先追求章节分布的平均。

### 2. 模板要能解释

任何抽查模板都必须在报告中解释：

- 抽了哪些范围
- 为什么抽这些范围
- 哪些范围没覆盖
- 因此结论还保留了哪些不确定性

### 3. 模板要能升级

`sampled` 的内部模板不应阻碍升级路径。正确方向是：

```text
sampled(template)
  -> chapter-full
  -> full-book
```

### 4. 连载与完结要分开考虑

完结作品和连载作品的“尾部窗口”含义不同：

- 完结作品：尾部窗口用于结局翻车检查
- 连载作品：尾部窗口更像“最新进度观察”

## 候选 coverage templates

下面这些模板是当前最值得考虑的内部抽查模板。

### `opening-100`

**目标**：快速确认开篇质量与早期大雷。

**适用场景**：

- 新书摸底
- 用户只想知道“前期能不能看”
- 连载作品刚开不久

**抽查思路**：

- 优先前 100 章左右
- 若总章节数不足 100，则按比例下调
- 对无明显章节结构的文本，退化为相近正文区间

**优点**：

- 最贴近社区最常见的快速扫法
- 最容易快速产出初判

**缺点**：

- 对后期翻车无保护
- 容易漏掉中后段关系反转与结局恶化

### `head-tail`

**目标**：同时覆盖开篇与结尾风险。

**适用场景**：

- 完结作品快速排雷
- 用户最担心“后期翻车”

**抽查思路**：

- 抽取前段窗口
- 抽取尾段窗口
- 中段不主动平均覆盖，除非存在额外高风险信号

**优点**：

- 更符合“首尾抽查”的社区习惯
- 能更早发现结尾恶化类大雷

**缺点**：

- 中段重大事件仍可能漏检
- 适合完结书，不完全适合长连载

### `head-tail-risk`

**目标**：在首尾窗口基础上，补关键风险热点。

**适用场景**：

- 完结长书
- 关系密集、风险密集作品
- 需要比 `head-tail` 更稳，但还不想直接全量

**抽查思路**：

- 先取开篇窗口
- 再取结尾窗口
- 再加入标题高风险批次 / 事件候选密集批次 / 补证问题密集批次

**优点**：

- 比纯首尾窗口更接近“决策导向抽查”
- 能利用现有 `risk-aware` 资产

**缺点**：

- 实现复杂度更高
- 报告必须清楚解释“为什么补这些点”

### `opening-latest`

**目标**：适配连载作品的“开篇 + 最新进度”摸底。

**适用场景**：

- 连载中作品
- 用户关心“最近有没有翻车”

**抽查思路**：

- 抽取开篇窗口
- 抽取最新章节附近窗口
- 必要时叠加热点风险批次

**优点**：

- 更贴近连载读者的真实行为

**缺点**：

- 无法替代对中段长期演化的完整判断

## 模板与现有能力的关系

### 与当前兼容执行层

当前可以把这些模板理解为 `sampled` 的内部实现演化方向；兼容执行层里它们仍主要落在 `economy` 路径，而不是与 `sampled` 平级的新模式。

### 与 `risk-aware`

`risk-aware` 更像模板中的一个增强策略，可以复用，但不应继续作为普通用户的第一理解层。

### 与 `chapter-full`

`chapter-full` 的重点是：

- 按章节尽量完整覆盖
- 若章节失败则退化为分段级全文扫描

它不是抽样模板，而是比 `sampled` 更高一层的覆盖模式。

### 与 `full-book`

`full-book` 是最终确认，不依赖抽查窗口。

## 报告层要求

未来无论采用哪种模板，报告都应新增或强化以下说明：

- 本次使用的抽查模板名称
- 实际覆盖的章节 / 区间
- 触发补扫的原因
- 未覆盖范围的提醒
- 哪类风险因此仍偏保守

推荐的报告表达结构：

```text
覆盖模板：head-tail-risk
已覆盖：第1-100章；第900-1000章；第420-440章（风险热点）
未覆盖：大部分中段常规章节
解释：本次优先验证开篇合法性、结局翻车风险与高风险热点
结论提示：若用户对中段关系演化极敏感，建议升级到 chapter-full / full-book
```

## 建议的落地顺序

### 第 1 步：先文档化

- 在产品文档中明确：`sampled` 是决策导向抽查，不等于均匀抽样
- 把内部模板先记为设计对象，而不是直接硬编码为用户主入口

### 第 2 步：先把模板概念落成可观察字段

- 在运行时先记录“当前使用了哪种覆盖模板”
- 即使先不改实际抽样逻辑，也先让报告与数据结构能容纳这个概念
- 当前仓库这一层已经落地，字段名固定为 `coverage_template`
- 当前会透传到：manifest 解析 -> `pipeline-state.json` -> `merged-report.json/md/html` -> DB `runs.jsonl`

### 第 3 步：再替换抽样策略实现

- 先从 `opening-100` 或 `head-tail` 开始
- 再演进到 `head-tail-risk`
- 最后再考虑连载专用的 `opening-latest`

当前代码基线已经完成这一步的第一轮落地：

- `opening-100`
- `head-tail`
- `head-tail-risk`
- `opening-latest`

都已经在 `sample_batches.mjs` 中成为可执行的覆盖模板；后续迭代重点是继续优化窗口大小、热点补刀逻辑与报告解释，而不是再把模板概念从头接一遍。

### 第 4 步：最后再决定是否对外暴露

- 若模板差异已经足够稳定、用户也能理解，再决定是否把其中一部分上升为用户可选项
- 在此之前，更适合由 skill / wizard 根据场景自动推荐

## 当前结论

当前最合适的策略不是继续把 `sampled` 理解成“均匀抽样模式”，而是：

- 对外把它讲成“快速摸底”
- 对内逐步把它演进成“社区习惯化决策抽查模板集合”

一句话说：

> `sampled` 的未来，不是更像统计抽样，而是更像一个真正懂扫书的人会怎么抽查。
