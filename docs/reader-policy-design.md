# 读者策略层设计（草案）

这份文档用来记录一个当前已经明确暴露出来、但还没有进入正式 schema 的方向：

> `saoshu` 不能长期把“社区默认扫书口径”误当成“所有读者都共享的一套固定价值判断”。

当前固定雷点、固定防御档、固定默认问法仍然有价值，但它们更适合作为 **默认 preset**，而不是唯一合法模型。

## 为什么要单独立项

现在的主流程已经能比较稳定地做这些事：

- 提取风险候选、郁闷点、事件候选、上下文引用
- 输出 `sampled / chapter-full / full-book` 三层 coverage-first 结果
- 把待补证风险、补证问题、关系边等资产沉淀下来

但它仍然把不少“用户策略”写成了“系统默认判断”。这会在下面这些用户上显得不够准确：

- 喜欢后宫文，但对 `百合` 不敏感
- 能接受 `百合`，但不能接受“关键女主被抢 / 共享 / 关系主位被改写”
- 能接受高争议题材，但只接受 `已确认`，不接受靠弱证据提前劝退
- 能接受郁闷桥段，但不能接受结局反转、后期翻车或关键女主立场背离

这说明当前问题不只是“雷点库还不够多”，而是：

- 证据层和裁决层没有拆干净
- 默认社区口径被过度固化
- 脚本开始承担本应由人机协同完成的个体化决策

## 设计目标

后续应把系统明确拆成三层：

### 1. 证据层

负责尽量稳定地产出：

- 事件候选
- 雷点 / 郁闷点 / 未证实风险
- `context_references`
- 覆盖不足说明
- 补证问题

这一层应尽量 **与具体读者偏好解耦**。

### 2. 策略层

负责把同一份证据投影到不同读者需求上。

这一层不应再只靠单个 `target_defense` 承担全部语义，而应逐步支持更细的读者策略对象。当前可先用 `reader_profile` 作为设计代号，后续再决定最终命名。

推荐把读者策略拆成下列维度，而不是继续把所有差异都硬塞进防御档：

- `hard_blocks`：一旦实锤就直接劝退的绝对禁区
- `soft_risks`：可以接受但必须显式披露的风险项
- `relation_constraints`：例如“可接受百合，但不能接受抢关键女主 / 共享关键女主 / 关键关系主位被改写”
- `scope_rules`：哪些角色算“关键女主”、哪些支线应进入结论
- `evidence_threshold`：多严格才允许从“待补证”升级为影响结论
- `coverage_preference`：愿意接受多大的未覆盖区，是否更偏保守升级
- `review_style`：更偏快结论，还是更偏先补证再下结论

### 3. 交互层

负责让用户、AI 和维护者高效协作。

这一层应优先支持：

- 用对话或向导澄清模糊偏好
- 在报告里明确“这是按哪个策略视角给出的建议”
- 必要时对同一份证据给出多个策略视角下的并排建议

## 脚本应该承担什么

脚本后续应明确回到 **基建与自动化**：

- 切批、扫描、归并、渲染、校验、同步、入库
- 把证据提纯、结构化、可复核化
- 为人机协同提供稳定的输入输出合同

脚本不应该继续承担这些职责：

- 假装替所有读者预设唯一价值排序
- 用越来越多硬编码分支替代偏好澄清
- 把“社区里常见的默认结论”误讲成“系统唯一真理”

一句话说：

> 脚本负责把事实和候选风险整理好；具体怎么裁决，应该更多交给“读者策略 + 人机协同复核”。

## 推荐推进顺序

### Phase 0：先记录，不急着改 schema

本轮先做三件事：

- 在架构文档里明确“证据层 vs 策略层”的边界
- 把当前固定防御/固定雷点口径标记为“默认 preset”
- 把新增需求记入路线图，避免后续继续把用户多样性问题塞回脚本细节里

### Phase 1：让报告先支持“策略视角”

优先级最高的不是新增一堆参数，而是让报告层可解释：

- 明确当前结论是基于哪套默认视角
- 当用户表达了额外偏好时，允许附加一个临时策略视角
- 把关键补证问题按该视角重排优先级

当前已落地的最小合同：

- manifest 支持 `reader_policy`
- `manifest_wizard` 默认会写出可编辑的 `reader_policy` 骨架
- `merged-report.json` / Markdown / HTML 已会展示“当前按哪种策略视角解释证据”
- `follow_up_questions` 已会受 `reader_policy` 影响，优先追问策略视角更敏感的风险
- `coverage_decision` 已会把 `evidence_threshold=strict` / `coverage_preference=conservative` 视为更保守的补证信号
- `scan-db` 已能通过 `coverage-decision-overview`、`policy-audit` compare、dashboard、trends 对策略差异做最小复盘

当前仍未落地的部分：

- `reader_policy` 直接参与主裁决逻辑
- 按不同策略视角并排输出多份结论
- 哪类风险最常因为策略差异而改变结论，当前仍缺更强的结构化复盘

### Phase 2：再决定是否进入 manifest / profile 文件

只有当策略维度已经稳定，才考虑把 `reader_profile` 正式进入 manifest 或独立 profile 文件。

此时更合适的方向是：

- 默认保留社区 preset
- 允许显式传入 profile
- 对同一份证据生成“默认视角 + 指定视角”的差异说明

### Phase 3：把 scan-db 的策略复盘从“可观察”补到“可解释”

当前 `scan-db` 已经能看出策略 preset、证据阈值、覆盖偏好和自定义开关的分布，也能按作品集合做 `policy-audit` compare。

下一步如果继续进入数据库层，重点不是统计“谁更敏感”，而是把复盘解释补强：

- 哪类策略最常触发升级覆盖
- 哪类风险最常因为策略差异而改变结论
- 哪些默认启发式经常和真实用户偏好冲突

## 当前阶段的临时原则

在正式进入策略层之前，先按下面原则维护：

- 现有 `target_defense`、固定雷点、固定郁闷规则继续保留
- 但统一把它们视为“默认社区 preset”，不是唯一真理
- 新需求若本质属于“读者偏好差异”，优先记入本设计文档或 roadmap
- 不用脚本硬编码新增一条“特殊用户模式名”来临时补锅

## 明确不做什么

当前不建议这样推进：

- 为每一种偏好单独膨胀一个新 `coverage_mode`
- 为每一种偏好都新增一个新的“防御档”
- 让脚本自动猜测用户价值观后直接替用户定论
- 为了追求全自动，把补证、追问、策略澄清全部压没

## 一个最小例子

下面这个例子足以说明“防御档”不是完整策略模型：

- 用户 A：能接受 `百合`，但不能接受关键女主被抢走
- 用户 B：不能接受 `百合`，但对一般关系拉扯不敏感

这两个人都可能自称“布甲”甚至“重甲”，但他们的真正决策边界并不一样。

因此，后续真正要做的不是继续细化“布甲到底算什么”，而是把：

- 风险证据
- 关系约束
- 读者策略

拆开表达，再通过人机协同把它们重新组合成结论。
