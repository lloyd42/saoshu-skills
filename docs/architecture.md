# 架构说明

## 核心定位

`saoshu` 的核心定位是 **skill 项目**，不是单纯的命令行工具项目。

这意味着：

- `skill` 是面向最终使用场景的能力本体
- `CLI` 是 `skill` 的子能力、自动化入口和扩展用法
- 独立使用 CLI 的用户通常更偏进阶，不应反向主导整体架构

项目的目标不是先做一个复杂工具箱，再额外包装成 skill；
而是先定义一个能力超集型 skill，再为其补充 CLI、校验链、扩展接口和自动化入口。

## 架构分层

### 1. 核心能力层

由 `packages/saoshu-harem-review` 承担。

这一层必须内聚，负责：

- 扫书主流程
- 规则与判定逻辑
- 输入输出契约
- 报告生成
- 无外部增强时的本地可运行能力

判断标准是：**没有这一层，skill 本身就不成立。**

### 2. 扩展能力层

包括但不限于：

- `packages/saoshu-term-wiki`
- `packages/saoshu-scan-db`
- 可选关系图、数据库、术语扩展等能力

这一层增强使用体验和结果可读性，但不应该成为 skill 可用性的前提。

判断标准是：**没有它，skill 仍然能运行；有了它，体验更强。**

### 3. 适配与增强层

包括外部命令、未来的 MCP、其他增强提供方。

这一层的定位是：

- 不是架构中心
- 不是主流程刚性依赖
- 只负责可插拔增强

当前 `packages/saoshu-mcp-enricher-adapter` 应视为增强适配层候选，而不是主流程核心。

## 为什么不以 CLI 为中心

如果以 CLI 为中心，很容易导致：

- 为命令组织而不是为能力组织架构
- 把大量进阶参数暴露给普通 skill 用户
- 扩展点优先服务工具链，而不是服务真实使用场景

而以 skill 为中心时：

- 用户看到的是能力，不是实现细节
- CLI 只是调用、自动化和高级用法入口
- skill 的渐进式披露天然可以覆盖大多数用户路径

## MCP 的合理定位

MCP 不是为了“多一个壳”，而应该只在下面场景中发挥价值：

- 外部能力明显强于本地兜底能力
- 需要统一接入多个外部提供方
- 返回结果可以结构化写回当前主流程
- 失败时可以自动回退到本地方案

因此，MCP 更适合被定义为：

- 一种可选增强实现
- 一种 provider / adapter
- 一种可替换的外部能力来源

而不适合被定义为：

- 主流程的一部分
- 用户必须理解的核心概念
- 没有它就无法运行的前置条件

## 内聚与解耦原则

### 应该内聚的部分

- 主流程编排
- 规则判定
- schema 与输出契约
- 本地 fallback 能力
- 主入口用户体验

### 适合解耦的部分

- 外部增强识别
- 术语/百科补充来源
- 数据库接入方式
- 可视化增强
- MCP / 外部 provider 适配

## 用户视角原则

从最终用户角度看，项目应满足：

- 默认可用
- 渐进增强
- 可按需配置
- 扩展失败不致命

因此用户层应优先看到：

- skill 能做什么
- 最小使用路径是什么
- 可选增强怎么启用

而不是先看到：

- MCP
- orchestrator
- adapter
- 内部模块边界

## 当前建议

- 保持 `saoshu-harem-review` 为唯一核心包
- 把 CLI 继续视为子入口，而不是项目主身份
- 把 MCP 继续降级为可选增强层，不扩大其存在感
- 后续新增功能时，优先判断它属于核心能力、扩展能力，还是适配层能力
