# 社区语境与产品对齐

这份文档用来记录 `saoshu` 当前对“扫书”社区语境、产品定位、模式边界、人工介入点与生态联动的统一理解，避免后续开发时只记得脚本实现，却忘了目标用户到底怎么思考问题。

## 为什么要有这份文档

- `saoshu` 面向的不是抽象“文本分析”用户，而是带着明确阅读边界、避雷需求、防御等级和社区黑话的网文读者。
- `yy小说吧`、`哀伤雪刃吧` 这类社区里的人工扫书爱好者，很可能就是本项目的最终用户之一。
- 这些社区里的叫法、判定顺序、犹豫点与升级行为，对模式设计、报告结构、术语系统和人机协同节点都有直接影响。

## 社区里的几个核心动作

下面这些定义不是官方术语词典，而是当前项目为了和真实社区语境对齐而采用的工作定义。

### 扫书

- 核心目标：快速判断“这本能不能看”。
- 关注重点：雷点、郁闷点、防御匹配、证据是否充分、是否需要继续补证。
- 输出形态：结论、证据、待补证风险、对不同防御等级的建议。
- 本项目的主定位就是这一层。

### 推书

- 核心目标：给某类读者推荐更可能适配的书。
- 关注重点：匹配度、题材偏好、爽点、风格、读者画像。
- 与扫书关系：推书可以消费扫书的风险判断，但不应反向主导扫书核心逻辑。

### 拆书

- 核心目标：分析作品结构、节奏、人物关系、桥段机制、争议点分布。
- 关注重点：作品为什么有效、哪里翻车、套路如何运作。
- 与扫书关系：拆书适合复用扫书沉淀的结构化中间层，但不等同于扫书结论。

### 同人 / 续写 / 衍生创作

- 核心目标：在尽量不违背原作连续性的前提下继续创作。
- 关注重点：人物设定、关系状态、时间线、禁区、未证实设定。
- 与扫书关系：扫书可以提供“创作防翻车约束”，但不直接承担创作生成本身。

## 当前产品定位

`saoshu` 当前最准确的定位，不是“小说分析工具”，而是：

> 一个面向读者决策的人机协同扫书 skill 系统。

它的核心任务是：

- 用最低认知成本给出初判
- 把风险、证据和不确定性说清楚
- 在需要时提供升级与人工介入路径
- 把一次次扫书沉淀成可复用资产

因此，项目默认不追求“全自动全理解”，而追求：

- 默认可用
- 渐进增强
- 关键争议可补证
- 长期使用可沉淀经验

这与 `docs/architecture.md` 中的 `skill-first` 定位保持一致。

## 默认社区口径不是唯一用户画像

当前社区里的扫书黑话、固定防御档和常见判定顺序，仍然很有价值，但它们更像：

- 一个默认社区 `preset`
- 一套高频读者群的工作语言
- 一个适合快速进入问题空间的起始模型

而不应被误讲成“所有读者都共享的一套固定偏好”。

现实中会持续出现这类差异：

- 有人喜欢后宫文，但对 `百合` 并不敏感
- 有人接受 `百合`，但不能接受关键女主被抢、共享或关系主位被改写
- 有人能接受争议题材，但只接受 `已确认`，不接受用弱证据直接劝退

因此后续产品不应只靠“再多加几条固定规则”来解决问题，而应逐步把：

- 证据抽取
- 默认社区 preset
- 读者个体策略

拆开表达。

这也是为什么后续要单独维护 `docs/reader-policy-design.md`：它记录的不是“又一个模式名”，而是“如何把用户偏好多样性从脚本硬编码里解耦出来”。

## 模式边界：执行层 vs 产品层

当前仓库里有两层模式概念，后续讨论时不要混为一谈。

### 1. 当前 coverage-first 用户语义层

- `sampled`：快速摸底
- `chapter-full`：章节级尽量完整
- `full-book`：整书最终确认

这是 coverage-first 的用户语义层。

### 2. 当前稳定执行层（兼容层）

- `economy`：当前抽样执行基线
- `performance`：当前高覆盖执行基线

这是当前真实已落地、已验证的执行基线，但不再是用户优先理解模式时的第一叙事层。

### 3. 当前两层的关系

当前代码仍以执行层为主，但运行时已接受 coverage-first 用户语义层的兼容字段：

- `sampled -> economy`
- `chapter-full -> performance`（当前已增加“章节失败 -> 分段级全文扫描”的真实差异）
- `full-book -> performance`（当前已增加“整书连续分段全文扫描”的真实差异）

这意味着现在仍不能把 `sampled / chapter-full / full-book` 误讲成三套完全独立的覆盖引擎；它们仍共享 `economy / performance` 主链，但 `chapter-full` / `full-book` 已有各自清晰的切批边界。

## 抽样 + 升级的正确位置

“抽样 + 升级” 应该是一条用户路径，而不是继续膨胀出新的模糊模式。

推荐心智模型：

```text
先快速摸底
  ↓
如果命中高风险、结论不稳、作品很重要
  ↓
升级到更高覆盖
  ↓
必要时继续人工补证
```

当前对应关系：

- 先走 `sampled`
- 需要更强确认时优先走 `chapter-full`
- 需要最终确认、争议复核或绕开章节前置时走 `full-book`
- 内部兼容执行层仍分别映射到 `economy / performance`

因此：

- `sample_level` / `sample_strategy` 更适合作为策略层或高级参数
- 用户主概念应当优先理解“覆盖级别”，而不是先理解“抽样花样”

## 社区里的章节抽查习惯

根据当前对贴吧人工扫书习惯的观察，很多用户并不是做“均匀随机抽样”，而是做带强烈目的性的章节抽查，例如：

- 先看前 100 章左右，快速确认题材、后宫合法性、第一批女主、文风与早期风险
- 首尾各看一部分，重点防后期翻车、关系反转、送女、背叛、死女、烂尾式恶化
- 对超长作品，在开篇摸底后，再补看结局段或关键大篇章，而不是平均每段都看一点

这说明一个重要产品事实：

> 社区用户做的不是“统计学抽样”，而是“决策导向抽查”。

因此后续的 `sampled` / 抽样层设计，应该优先考虑：

- 开篇窗口
- 尾部窗口（完结作品）
- 关键风险窗口
- 必要时的中段补刀

而不应只用纯 `uniform` 语义去描述用户体验。

### 对模式设计的启发

- `sampled` 更像“社区习惯化摸底”，不是纯技术抽样。
- `chapter-full` 才是“按章节尽量完整地扫”。
- `full-book` 是“最终确认”，不再依赖抽查窗口。

### 对内部实现的启发

后续如果继续演进抽样层，优先考虑把“贴吧习惯”做成 `sampled` 的内部 profile / template，而不是再额外膨胀新的用户可见模式名。

可考虑的内部模板示例：

- `opening-100`：优先前 100 章
- `head-tail`：首尾窗口各取一段
- `head-tail-risk`：首尾窗口 + 风险热点批次
- `opening-latest`：连载作品优先前段 + 最新进度附近

这些模板后续若落地，报告中应明确告诉用户：

- 实际扫了哪些章节范围
- 为什么选这些范围
- 哪些范围仍未覆盖
- 因此哪些风险结论仍需保守看待

## 用户最小旅程与人工介入点

### 默认旅程

```text
manifest 向导
  -> scan / run_pipeline
  -> 打开 HTML 报告
  -> 仅在争议点介入
```

新手默认应先看到：

- 结论
- 关键证据
- 未证实风险
- 最关键的补证问题

而不是一长串高级参数。

### 人工 / AI 介入点 1：章节 assist

- 当章节识别失败或低置信时，不应整条链路直接报废。
- 系统应生成协作包，让当前 AI / 人工回填章节边界后继续运行。
- 这不是降级体验，而是人机协同的关键救援口。
- 这个介入点主要服务 `chapter-full`；如果你的目标只是最终确认，也可以直接走 `full-book`，让整书连续分段全文扫绕开章节前置。

### 人工 / AI 介入点 2：review -> apply

- 系统先产出待复核事件与上下文
- 人工或 AI 再把条目标为 `已确认 / 排除 / 待补证`
- 回填后再 `merge`

这条链路是扫书可信度的核心，不是附加功能。

### 人工 / AI 介入点 3：mode-diff 治理

- `mode-diff` 的产品意义，是回答“当前快速摸底够不够用”。
- 单次差异主要用于提示“回退完整复核”或“增强当前抽样”。
- 只有多本书长期落在灰区，才值得讨论中档模式或更细分覆盖层。

## feedback loops 的定位

下面这些能力应当继续保留，但其角色应明确为“辅助层 / 资产层”，而不是主扫描引擎本体：

- `keyword_candidates -> keyword_promotions -> keyword_rules`
- `alias_candidates -> alias_promotions -> alias_map`
- `risk_question_candidates -> risk_question_promotions -> risk_question_pool`
- `relation_candidates -> relation_promotions -> relationship_map`

它们的主要价值是：

- 热点提示
- 人工 review 排序
- 长期沉淀社区知识与项目经验

而不是继续堆成越来越复杂的自动启发式主入口。

同理，feedback loop 也不应被拿来偷偷替代“用户策略层”。它们更适合沉淀：

- 高频风险词
- 关系约束线索
- 常见补证问题
- 社区默认 preset 的经验资产

但最终如何裁决，仍应允许被不同读者策略重新解释。

## 如果与其它项目联动，扫书最该提供什么

扫书项目最适合提供的是“可信上游资产”，而不是包打天下。

### 给推书

- 风险过滤
- 防御等级匹配
- 已确认雷点与待补证风险
- 推荐解释里的避雷依据

### 给拆书

- 章节切分结果
- 事件链与争议节点
- 别名归一化后的人物关系
- 高风险桥段分布

### 给同人 / 续写

- 已确认角色设定
- 当前关系状态
- 关键事件年表
- 禁区 / 高风险设定提示
- 未证实设定空白区

### 给社区知识库 / 术语系统

- 高频风险词
- 社区别名与黑话口径
- 常见补证问题
- 不同社区用法差异

## 架构原则：扫书核心只做可信判断层

为了未来能和推书、拆书、同人 / 续写等项目联动，当前核心包应坚持下面的边界：

- `saoshu-harem-review` 负责：判断、证据、不确定性、结构化输出、本地 fallback
- 推书 / 拆书 / 创作辅助等，适合以后做成独立 package 或独立 skill
- 数据库、术语、外部增强继续放在扩展层 / 适配层，不反客为主

一句话说：

> 扫书项目不需要自己做完所有事，但应当成为其它网文项目最可信的上游判断层。

## 当前已确认的下一步方向

在 2026-03-07 这一轮讨论后，当前最值得继续推进的方向是：

1. 继续明确 coverage-first 的模式边界与用户语义
2. 真正实现“无章节时自动退化为分段级全文扫描”
3. 强化章节 assist、review/apply、mode-diff 的人机协同体验
4. 逐步补社区术语、社区案例与生态契约，而不是继续深抠抽样表面参数
5. 把固定防御档与固定雷点口径收回到“默认社区 preset”，为后续读者策略层留出清晰边界
