# 扫书执行协议

## 1) 锁定任务边界
- 仅处理“男主开后宫、女主全为女性”的后宫文。
- 若作品不满足后宫边界，立即标注`不适用本技能`并给出原因。
- 若用户未给防御等级，默认输出`重甲/布甲/轻甲/低防`四档建议，并补一句询问是否要精确到单一防御。

## 2) 先收集，再结论
- 先提取要素：题材、女主名单、第一女主、是否重生/穿书/同人、前世线、关键情节、结局。
- 信息不足时不得硬判：
  - 标注`已确认`、`高概率`、`待补证`三种证据级别。
  - 对`待补证`给最小补充问题（最多3条）。
  - `未知待证`仅作为兼容旧字段的内部别名，不作为新的对外主术语。

## 3) 判定顺序（固定）
1. 后宫合法性检查（不合法则结束）。
2. 六大雷点检查（见 `references/rules.md`）。
3. 郁闷点逐条检查并标注严重度+最低可抗防御（见 `references/rules.md`）。
4. 生成防御匹配与结论（见 `references/output.md`）。

## 4) 冲突处理原则
- 同一情节命中多条规则时，按“更严重”计。
- 只要命中任一雷点：
  - 除`神防之上`外默认`劝退`。
  - `神防`对非绿帽雷可给“可看但重雷警告”。
- 对“可能绿帽/可能背叛/可能死女”但证据不足的情况，必须单列`未证实风险`。

## 5) 输出要求
- 必须使用 `references/output.md` 的模板。
- 禁止模糊词替代结论（如“可能还行”）。必须落到：可看 / 慎入 / 劝退。
- 每个郁闷点必须附：具体情节 -> 程度 -> 最低可抗防御 -> 证据级别。

## 6) 语气与质量
- 客观、去情绪化、不给作者或角色做价值审判。
- 不编造剧情；不确定就明确说不确定。
- 优先可执行价值：让不同防御等级读者直接据此决策。

## 7) 百万字长书模式（强制分批）
- 当全文过长无法一次处理时，必须启用 `references/long-book.md` 的分批协议。
- 禁止跳批次；禁止只看开头和结尾直接下结论。
- 每批输出必须沉淀到“证据账本”（雷点/郁闷点/人物关系变化/未证实风险）。
- 最终结论只能基于“归并后总账本”，不能基于单批印象。
- 角色识别、关系抽取、标签识别优先使用外部工具（如 MCP）；本地脚本只做兜底。

## 8) 快速工作流（可直接复用）
- 统一入口（推荐）：`scripts/saoshu_cli.mjs`
  - 扫书：`node scripts/saoshu_cli.mjs scan --manifest <manifest.json>`
  - 术语：`node scripts/saoshu_cli.mjs wiki --term <词>`
  - 统计：`node scripts/saoshu_cli.mjs db overview --db <db-dir>`
  - 对比：`node scripts/saoshu_cli.mjs compare --db <db-dir> --output-dir <out-dir>`
- Step A: 复述用户目标与防御等级。
- Step B: 判断是否进入长书模式。
- Step B.1: 先对照 `references/architecture/overview.md` 与 `references/schemas/*.json`，确认本次改动不破坏全局输出架构。
- Step C: 若是长书模式，先用 `scripts/scan_txt_batches.mjs` 生成批次草案。
- Step D: 用 `scripts/enrich_batches.mjs` 做元数据增强（外部工具优先，失败回退本地）。
- Step E: 按 `references/long-book.md` 人工复核关键风险。
- Step F: 用 `scripts/batch_merge.mjs` 生成 `json/md/html` 报告并输出结论。
